CD na teoria: do push ao servidor

CI, Continuous Delivery e Continuous Deployment são palavras parecidas que significam coisas distintas. Estratégias de deploy e a anatomia de um workflow GitHub Actions.

inteli

CI, Continuous Delivery e Continuous Deployment, três coisas distintas

O mercado usa as três siglas de forma intercambiável e isso confunde. A distinção importa porque define onde termina a automação e onde começa a responsabilidade humana. O podcast de auto-estudo toca nesse ponto quando fala em "cultura de confiança no pipeline".

Continuous Integration

Todo push integra com a branch principal. Pipeline roda build e testes. Objetivo: detectar conflito e defeito cedo. Entrega: artefato verificado. Nenhum deploy acontece aqui.

Continuous Delivery

O artefato verificado está sempre pronto para ir a produção. O deploy para produção pode ser acionado manualmente, por um botão ou aprovação. O pipeline garante que o código está deployável, não que ele foi deployado.

Continuous Deployment

Todo push que passa no pipeline vai para produção sem aprovação humana. Requer maturidade alta: feature flags, rollback automatizado, observabilidade sólida. É o que vamos construir hoje.

O que os engenheiros do podcast disseram

Empresas com menos maturidade começam em Continuous Delivery: pipeline automatizado, deploy manual com aprovação. Migram para Continuous Deployment quando confiam no teste e no rollback. Forçar CD sem essa confiança resulta em incidentes frequentes e equipe revertendo para deploy manual na mão.

Estratégias de deploy, como o código chega ao usuário

Deploy não é apertar um botão. É uma operação que tem estratégia: quanto do tráfego muda, com que velocidade, com que plano de volta. Três estratégias dominam o mercado e diferem pelo nível de risco aceito durante a transição.

Rolling Update

Instâncias são substituídas uma a uma. A nova versão entra enquanto a velha ainda serve parte do tráfego. Zero downtime, mas há momento com duas versões no ar ao mesmo tempo.

✓ Simples de implementar
✗ Difícil de rollback rápido

Blue/Green

Dois ambientes idênticos. Blue serve produção, Green recebe a nova versão. Quando Green está pronto, o load balancer vira a chave. Rollback = virar a chave de volta.

✓ Rollback instantâneo
✗ Custo: dois ambientes simultâneos

Canary

Nova versão recebe uma fatia pequena do tráfego (5%, 10%). Métricas são monitoradas. Se estiver ok, o tráfego aumenta gradualmente até 100%. Rollback só afeta a fatia.

✓ Risco progressivo
✗ Exige observabilidade madura

Para a Sprint de hoje

O Hello World usa Rolling simples: uma instância EC2, CloudFormation atualiza, SSH puxa o código novo. É o ponto de entrada adequado. Blue/green e canary são temas da Sprint 03, quando a aplicação real tiver usuários reais e a tolerância a downtime for zero.

GitHub Actions, anatomia de um workflow

O tutorial do auto-estudo mostrou o passo a passo entre os minutos 14 e 35. Aqui está a estrutura que ficou implícita no vídeo. Cinco conceitos, cada um com papel distinto. Saber onde cada peça encaixa é o que separa quem copia YAML de quem escreve YAML.

# .github/workflows/deploy.yml
name: Deploy to EC2        # nome visível no GitHub UI

on:                        # EVENTO que dispara o workflow
  push:
    branches: [main]

jobs:                      # conjunto de JOBs
  build-and-deploy:        # nome do job
    runs-on: ubuntu-latest # RUNNER (máquina virtual)

    steps:                 # sequência de STEPs
      - uses: actions/checkout@v4  # ACTION do marketplace

      - name: Run tests    # step inline
        run: npm test      # comando shell

      - name: Deploy
        uses: appleboy/ssh-action@v1
        with:              # parâmetros da action
          host: ${{ secrets.EC2_HOST }}
          key:  ${{ secrets.EC2_SSH_KEY }}
          script: pm2 restart all

Os cinco pilares

  1. Workflow: o arquivo YAML inteiro. Um repo pode ter vários.
  2. Event (on:): o que dispara o workflow — push, PR, schedule, manual.
  3. Job: conjunto de steps que roda numa mesma máquina. Jobs em paralelo por padrão.
  4. Runner: a VM que executa o job. GitHub hospeda ou você traz o seu.
  5. Step: unidade atômica — um comando shell (run:) ou uma Action (uses:).

Actions são pacotes reutilizáveis do marketplace. actions/checkout@v4 já tem mais de 4 bilhões de execuções registradas.

Events, contexts e o marketplace de Actions

O que dispara um workflow define todo o comportamento do pipeline. Misturar eventos errados com o workflow certo gera CI que deploya quando não deve, ou deploy que não acontece quando deveria.

Mais comum

push / pull_request

Deploy em push para main. Testes em qualquer PR. Filtra branch com branches:.

Agendamento

schedule

Cron no GitHub. Útil para DAST noturno, limpeza de staging, relatórios periódicos.

Aprovação humana

workflow_dispatch

Botão manual no GitHub UI. Permite parâmetros. Útil para deploy com confirmação explícita.

Encadeamento

workflow_run

Dispara após outro workflow terminar. CD que espera CI verde antes de começar.

Contextos e expressões

O objeto github expõe metadados do evento: github.sha (hash do commit), github.actor (quem fez o push), github.ref (branch). secrets expõe os segredos do repo. env expõe variáveis de ambiente do job. Esses três contextos cobrem 95% do que aparece em workflows reais. O marketplace tem mais de 20.000 Actions publicadas — para quase tudo já existe uma Action pronta.

CloudFormation, infraestrutura como código na AWS

CloudFormation é o serviço de IaC nativo da AWS. Você descreve a infraestrutura que quer num arquivo YAML (ou JSON), e a AWS cria, atualiza ou deleta os recursos automaticamente. A unidade de trabalho é o stack: um conjunto de recursos que nascem e morrem juntos.

O que é um template

Arquivo YAML com seções padronizadas. Define quais recursos criar (EC2, S3, RDS, IAM role...) e como eles se conectam. Versionado no repositório junto do código.

O que é um stack

Instância do template na AWS. Você cria um stack, a AWS lê o template e provisiona os recursos. Atualizar o template e reaplicar faz o CloudFormation calcular o diff e executar só o que mudou.

Por que usar no CD

Infraestrutura reproduzível: o mesmo template em dev, staging e produção garante que o ambiente que testamos é o ambiente que deployamos. Fim de "funciona na minha máquina".

Relação com o artigo de auto-estudo

O artigo da AWS publicado em março de 2022 usa exatamente esse padrão: um template CloudFormation cria a instância EC2 com Security Group configurado, e o workflow GitHub Actions roda aws-cloudformation-github-deploy para criar ou atualizar o stack antes de fazer o deploy da aplicação. Vamos ver cada parte desse template agora.

Template CloudFormation para EC2, linha por linha

Esse é o template que o seu projeto deve ter em cloudformation/ec2.yml. Cada seção tem uma responsabilidade distinta. Leia o YAML e as anotações ao mesmo tempo.

AWSTemplateFormatVersion: '2010-09-09'
Description: EC2 instance, Hello World CD demo

Parameters:
  KeyName:
    Type: AWS::EC2::KeyPair::KeyName
    Description: Par de chaves SSH para acesso
  MyIPAddress:
    Type: String
    Description: IP autorizado para SSH (CIDR /32)

Resources:
  WebServerSG:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: Allow HTTP and SSH
      SecurityGroupIngress:
        - IpProtocol: tcp
          FromPort: 22
          ToPort: 22
          CidrIp: !Sub "${MyIPAddress}/32"
        - IpProtocol: tcp
          FromPort: 80
          ToPort: 80
          CidrIp: 0.0.0.0/0

  WebServer:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: ami-0c02fb55956c7d316   # Amazon Linux 2
      InstanceType: t2.micro
      KeyName: !Ref KeyName
      SecurityGroups:
        - !Ref WebServerSG
      UserData:
        Fn::Base64: |
          #!/bin/bash
          yum update -y
          yum install -y nodejs git
          npm install -g pm2

Outputs:
  PublicIP:
    Value: !GetAtt WebServer.PublicIp
    Description: IP publico da instancia

Seções e o que fazem

  1. Parameters: valores que o workflow passa na hora de criar o stack. KeyName e MyIPAddress vêm dos GitHub Secrets — jamais hardcoded.
  2. Resources: o coração do template. WebServerSG abre porta 80 para o mundo e porta 22 só para o IP autorizado. WebServer é a instância EC2.
  3. UserData: script que roda uma vez ao criar a instância. Instala Node, git e PM2. O deploy da app vem depois via SSH.
  4. Outputs: PublicIP é exportado para que o step seguinte do workflow saiba onde fazer o SSH.

!Ref e !GetAtt são funções intrínsecas do CloudFormation para referenciar outros recursos do mesmo template.

Credenciais AWS no GitHub Actions, sem expor nada

Para o workflow falar com a AWS ele precisa de credenciais. A forma errada (e muito comum) é colocar Access Key e Secret no YAML. A forma certa é usar GitHub Secrets combinados com IAM de mínimo privilégio.

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Step que autentica o runner na AWS
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: us-east-1

      # A partir daqui, todos os steps têm acesso à AWS
      - name: Deploy CloudFormation stack
        id: cfn
        uses: aws-actions/aws-cloudformation-github-deploy@v1
        with:
          name: hello-world-stack
          template: cloudformation/ec2.yml
          parameter-overrides: >-
            KeyName=${{ secrets.EC2_KEY_NAME }},
            MyIPAddress=${{ secrets.MY_IP_ADDRESS }}

Como configurar os Secrets

  1. No repositório: Settings → Secrets and variables → Actions
  2. Criar: AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, EC2_KEY_NAME, MY_IP_ADDRESS, EC2_SSH_KEY
  3. O IAM User da AWS deve ter somente as permissões necessárias: cloudformation:*, ec2:*. Nunca usar o usuário root.
  4. ${{ secrets.X }} nunca aparece nos logs, o GitHub mascara automaticamente.

O step aws-cloudformation-github-deploy cria o stack se não existe ou atualiza se existir — idempotente por design.

O workflow completo: do push ao Hello World no ar

Baseado diretamente no artigo da AWS. Cada step tem um papel. Leia os comentários junto com as anotações.

name: Deploy Hello World to EC2
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: us-east-1

      - name: Provision EC2 via CloudFormation
        id: cfn
        uses: aws-actions/aws-cloudformation-github-deploy@v1
        with:
          name: hello-world-stack
          template: cloudformation/ec2.yml
          parameter-overrides: >-
            KeyName=${{ secrets.EC2_KEY_NAME }},
            MyIPAddress=${{ secrets.MY_IP_ADDRESS }}

      - name: Deploy app via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ steps.cfn.outputs.PublicIP }}
          username: ec2-user
          key: ${{ secrets.EC2_SSH_KEY }}
          script: |
            cd /home/ec2-user/app \
              || git clone ${{ github.server_url }}/${{ github.repository }} /home/ec2-user/app
            cd /home/ec2-user/app
            git fetch --all
            git reset --hard origin/main
            npm install --production
            pm2 restart hello-world || pm2 start index.js --name hello-world

5 steps, 5 responsabilidades

  1. checkout: clona o repo no runner para ter acesso ao template e ao código.
  2. configure-aws-credentials: autentica o runner na AWS usando os Secrets. Válido para os steps seguintes.
  3. aws-cloudformation-github-deploy: cria ou atualiza o stack. O output PublicIP fica em steps.cfn.outputs.PublicIP.
  4. appleboy/ssh-action: abre conexão SSH para o EC2 e executa o script de deploy.
  5. script SSH: clona o repo se for a primeira vez ou reseta para a branch main, instala dependências e (re)inicia o processo com PM2.

Nenhuma credencial ou IP aparece em texto puro no YAML.

Hello World: deploy automático no EC2

  1. Criem um repositório público (ou usem o da Sprint). Adicionem um arquivo index.js com um servidor HTTP simples que responde Hello World na porta 80. Commitem e façam o push.
  2. Criem a pasta cloudformation/ no repositório e adicionem o arquivo ec2.yml com o template que vimos no slide anterior. Ajustem o ImageId para a região us-east-1.
  3. No GitHub, vão em Settings → Secrets and variables → Actions e cadastrem os cinco secrets: AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, EC2_KEY_NAME, MY_IP_ADDRESS e EC2_SSH_KEY. Usem as credenciais do sandbox do Inteli.
  4. Criem o arquivo .github/workflows/deploy.yml com o workflow completo do slide anterior. Committem, façam o push para main e observem o workflow rodar em Actions.
  5. Quando o workflow ficar verde, abram o IP público no navegador. Hello World deve aparecer. Façam uma segunda mudança no index.js, façam o push e confirmem que o update chegou automaticamente.

Se o step de SSH falhar com "Connection refused", a instância ainda está inicializando. Adicionem um step de espera com sleep 30 antes do SSH. Se o CloudFormation falhar, leiam o evento de erro no console da AWS — 90% dos casos é permissão de IAM faltando.

Debrief D, o que sempre trava na primeira vez

Três problemas se repetem em toda turma que faz esse hands-on pela primeira vez. Mapeio aqui para vocês não perderem tempo com o mesmo obstáculo.

1, IAM sem permissão

O step do CloudFormation falha com AccessDenied. Causa: IAM User criado sem as policies de EC2 e CloudFormation. Solução: no console AWS, adicionar as managed policies AmazonEC2FullAccess e AWSCloudFormationFullAccess ao usuário.

2, SSH: "Permission denied"

A chave privada no Secret está errada ou com formatting quebrado. Chave SSH precisa ser a versão PEM completa, incluindo -----BEGIN RSA PRIVATE KEY----- e -----END-----. Copiem usando cat key.pem, não pelo explorador de arquivos.

3, Porta 80 sem resposta

Node está rodando mas na porta 3000, não 80. Confiram o index.js (porta 80 requer sudo no Linux) ou mudem o Security Group para liberar a porta que o Node está ouvindo e ajustem a URL de acesso.

A segunda iteração é o teste de verdade

O primeiro push prova que o pipeline funciona. O segundo push — mudando uma linha do Hello World — prova que o CD funciona. Quem não fez o segundo push não terminou o exercício. A automação precisa ser observada em ação pelo menos duas vezes para deixar de parecer mágica.

Governança e segurança no CD

Secrets management, deployment gates com aprovação humana, estratégia de rollback. O que separa um pipeline funcional de um pipeline confiável.

inteli

Secrets management, nenhuma credencial no código

Secret hardcoded é o erro mais caro e mais desnecessário da engenharia de software. O GitHub já tem o mecanismo certo, basta usar. A tabela abaixo mostra o que vai onde.

❌ Nunca faça

Colocar AWS_ACCESS_KEY_ID=AKIAIOSFODNN7EXAMPLE direto no .yml, no .env commitado, ou em variável de ambiente no Dockerfile. O TruffleHog da Aula 05 captura exatamente isso.

✓ GitHub Secrets

Credenciais de curta duração, IAM keys, chave SSH, tokens de API. Escopo por repositório ou organização. Mascarado nos logs automaticamente.

✓ AWS Secrets Manager

Secrets que a aplicação em execução precisa em runtime (credencial de banco, API key de terceiro). A instância EC2 busca o secret via SDK — nada em variável de ambiente no processo.

↗ OIDC (melhor prática)

Em vez de IAM Keys, o runner GitHub se autentica na AWS via OpenID Connect. Sem Access Key, sem Secret Key, sem rotação. O configure-aws-credentials suporta essa modalidade desde 2022.

Para hoje, IAM Keys nos Secrets é suficiente. OIDC é o próximo passo natural para qualquer projeto que vai para produção de verdade.

Deployment gates, quando a automação para e pede um humano

Continuous Deployment total (sem aprovação) é adequado para staging e para aplicações internas. Para produção com dados de usuário real, uma pausa para aprovação humana é governança, não burocracia. O GitHub tem isso nativo via Environments.

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test

  deploy-staging:
    needs: test
    runs-on: ubuntu-latest
    environment: staging   # ambiente sem aprovação
    steps:
      - name: Deploy para staging
        run: echo "deploy staging"

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production  # ambiente COM aprovação obrigatória
    steps:
      - name: Deploy para producao
        run: echo "deploy prod"

Como configurar o gate

  1. Vá em Settings → Environments → production
  2. Ative Required reviewers e adicione os aprovadores
  3. O workflow pausa no job deploy-production e manda e-mail para os revisores
  4. Após aprovação, o deploy continua automaticamente

Você pode adicionar regras como "aguardar 30 minutos após o último push" para evitar deployes precipitados em horário de pico.

Environments são gratuitos em repositórios públicos.

Rollback, o deploy que desfaz o deploy

Todo time que faz CD precisa saber como voltar atrás mais rápido do que demorou para ir pra frente. Sem rollback definido antes do incidente, o rollback vira improvisação sob pressão. Três mecanismos, cada um para um contexto.

Rerun do workflow anterior

No GitHub Actions, cada run tem um botão "Re-run". Ré-executar o último deploy que funcionou leva o código da versão boa de volta ao servidor. Simples, sem precisar de branch extra.

CloudFormation rollback

Se o stack update falhar, CloudFormation reverte automaticamente para o estado anterior por padrão. Pode-se também usar aws cloudformation continue-rollback-execution manualmente.

Feature flag

O código novo está no ar mas a feature está desligada por uma flag. Rollback = desligar a flag no painel, sem novo deploy. Exige mais arquitetura mas é o rollback mais rápido de todos.

A regra do runbook

O rollback precisa estar documentado antes do incidente. Um runbook de 4 linhas — "como voltar atrás em 5 minutos" — vale mais que qualquer documentação de arquitetura produzida com calma. O podcast de DevOps chama isso de "design para falha": aceitar que o deploy vai dar errado e preparar a saída antes da entrada.

Checagem 1 de 2, CI vs CD

Um time tem CI configurado. Todo push para main roda os testes e gera um artefato Docker. Para subir para produção, alguém ainda precisa clicar um botão no painel. Como se classifica esse processo corretamente?

  • Continuous Deployment, porque o deploy ocorre automaticamente após os testes
  • Continuous Delivery, porque o artefato está sempre pronto mas o deploy para produção requer ação humana
  • Continuous Integration, porque o foco é a integração contínua do código na branch principal
  • Rolling Deployment, porque o processo é incremental e controlado

Checagem 2 de 2, CloudFormation

Você quer que o step de SSH do seu workflow saiba o IP público da instância EC2 criada pelo CloudFormation. Qual é a forma correta de obter esse dado dentro do workflow?

Checklist do pipeline CD, o que um deploy maduro tem

Cinco critérios para avaliar se o pipeline de CD do seu projeto está pronto para algo além de sandbox. Não precisam estar todos verdes hoje. Precisam ter justificativa para os que ainda estão em aberto.

1, IaC versionado

Template CloudFormation (ou equivalente) no repositório, ao lado do código. Infra reproduzível em qualquer conta AWS.

2, Zero segredo no código

Nenhuma credencial, chave ou token em texto puro no repositório, no workflow ou no Dockerfile.

3, Pipeline idempotente

Rodar o workflow duas vezes no mesmo commit produz o mesmo resultado. CloudFormation e o script SSH já garantem isso.

4, Rollback documentado

Runbook de quatro linhas: como voltar à versão anterior em menos de 5 minutos. Escrito, testado, linkado na wiki do projeto.

5, Gate de produção
